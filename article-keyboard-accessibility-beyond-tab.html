Keyboard Access

Keyboard Accessibility Beyond the Tab Key

Keyboard accessibility is more than reaching every link. It includes predictable order, correct interaction patterns, visible focus, and a reliable way to get out of every component.

Tab Order Should Match the Visual Workflow

Keyboard focus should move through the page in a logical order that matches how the content is presented. Unexpected jumps can make a workflow feel broken even when every control is technically reachable.

Real Example: Menus and Modals

A modal should move focus inside when it opens, keep focus contained while open, close with Escape, and return focus to the button that opened it. A menu may need arrow-key navigation depending on the pattern being used.

  • Test Tab, Shift+Tab, Enter, Space, Escape, and arrow keys where relevant.
  • Confirm focus never disappears behind overlays or disabled states.
  • Make sure users can complete the full task without a mouse.

Keyboard Traps Are High-Impact Barriers

If a user can tab into a component but cannot leave it, the experience can become completely blocked. Keyboard traps should be treated as urgent because they prevent task completion.
